01

Automatisation des sauvegardes réseau

Mise en place d'un système complet de sauvegarde automatique des configurations des équipements réseau (switchs Aruba et Moxa) ainsi que des sauvegardes vCenter, sur un serveur dédié Linux Debian. Cette mission couvre l'ensemble de la chaîne : serveur, sécurisation des accès, scripts et planification.

Mise en place du serveur Debian

  • Installation de Debian 13 sur un serveur dédié aux sauvegardes
  • Configuration réseau : VLANs, adressage IP, règles pare-feu
  • Création d'un environnement chroot par type de sauvegarde (isolation des utilisateurs)
  • Configuration du serveur SFTP avec authentification dédiée
  • Principe retenu : les équipements poussent leurs configs vers le serveur (approche Push)

Script Bash — Switchs Aruba

  • Développement d'un script Bash de sauvegarde automatique des configurations
  • Intégration de Keychain pour la persistance de l'agent SSH entre redémarrages
  • Utilisation de la variable SSH_OPTS pour forcer les algorithmes et le mode BatchMode
  • Migration du script vers /etc/ pour une structure système robuste
  • Planification via Crontab (exécution toutes les 20h)
  • Debugging via les logs système : résolution des problèmes de clés SSH et d'authentification

Switchs Moxa — MXview

  • Abandon de la méthode par script (non supportée sur les switchs Moxa)
  • Mise à jour de l'outil MXview développé par Moxa
  • Configuration et planification de jobs de sauvegarde via MXview
  • Analyse des logs : résolution de l'erreur "authentication failed" (mauvais mot de passe global)

Sauvegardes vCenter (Push SFTP)

  • Configuration du vCenter pour exporter automatiquement ses sauvegardes en mode Push SFTP
  • Création d'un compte utilisateur de service dédié avec accès SFTP uniquement
  • Configuration du fichier sshd_config avec un bloc Match Group ciblé :
  • ForceCommand internal-sftp — interdit toute commande shell
  • ChrootDirectory — confine l'utilisateur dans le répertoire de sauvegarde
  • Désactivation du tunneling, du forwarding agent et de X11 (durcissement maximal)
  • Diagnostic via journalctl -u ssh : résolution de l'échec d'authentification par mot de passe

Difficultés rencontrées

  • Les switchs Aruba ne supportaient pas les clés SSH pour les connexions sortantes → changement d'approche
  • L'agent SSH étant temporaire (lié à la session), utilisation de Keychain pour la persistance
  • Les switchs Moxa ne supportaient pas le scripting → recours à MXview
  • Échec SFTP vCenter : le serveur SSH exigeait une clé publique alors que vCenter utilisait un mot de passe

Résultats obtenus

  • Sauvegardes automatiques opérationnelles sur les switchs Aruba et Moxa
  • Sauvegardes vCenter fonctionnelles en Push SFTP sécurisé
  • Serveur Debian isolé et durci (chroot, nologin, pas de shell)
  • Procédure documentée et démontrée à l'équipe

02

Gestion de flotte mobile — SOTI MobiControl

Déploiement et administration complète de la solution MDM (Mobile Device Management) SOTI MobiControl pour la gestion des smartphones et tablettes des agents de terrain (lutte anti-fraude, magasin Tram...). La mission couvre la création des profils, l'enrôlement des appareils et la sécurisation en mode Kiosque.

Création de profils et déploiement téléphones

  • Création de 3 profils distincts adaptés aux besoins métiers (profils personnalisés et génériques)
  • Chaque profil dispose de son propre ensemble d'applications métier
  • Déploiement des profils sur les téléphones agents de terrain
  • Résolution de dysfonctionnements applicatifs : paquets manquants (Google Photos), app obsolète (PingTools 2022)
  • Diagnostic : identification des noms de paquets Play Store et incompatibilités Samsung Galaxy A17

Enrôlement des tablettes métiers

  • Test de la méthode d'enrôlement préconisée par le groupe : 7 tapes sur l'écran d'usine pour activer le lecteur QR Code
  • Scan du QR Code groupe → enrôlement réussi des tablettes de test
  • Tests approfondis suite à l'installation (risques de conflits MDM/système natif)
  • Enrôlement et configuration de 7 nouvelles tablettes métier
  • Création de profils spécifiques par domaine métier : Magasin Tram (gestion stock pièces de rechange)
  • Tests d'enrôlement sur tablette Samsung Galaxy Tab Active5 (sans IMEI/Wi-Fi uniquement)

Mode Kiosque — Sécurisation

  • Déploiement du mode Kiosque pour restreindre la navigation aux applications autorisées
  • Identification d'une faille de contournement : sélection de texte → "Recherche Web" → navigateur libre
  • Recherche et application de correctifs pour verrouiller l'interface Chrome
  • Mise en place d'une politique de mise à jour automatique nocturne (00h00–03h00)
  • Investigation sur le blocage du contrôle à distance sans Wi-Fi

Difficultés rencontrées

  • Applications non fonctionnelles après déploiement (paquets Play Store non supportés)
  • Conflits potentiels entre MDM et système natif des tablettes
  • Faille de sécurité identifiée dans le mode Kiosque sur Chrome

Résultats obtenus

  • Flotte de smartphones et tablettes entièrement gérée via SOTI MobiControl
  • Mode Kiosque sécurisé et faille Chrome corrigée
  • Profils métiers adaptés déployés sur tous les appareils
  • Mises à jour automatiques planifiées hors heures de service

03

Diagnostic et sécurité réseau

Interventions de diagnostic sur des dysfonctionnements réseau critiques et réalisation d'audits de sécurité terrain sur les deux principaux sites de l'entreprise, dans le cadre d'une démarche de hardening de l'infrastructure.

Diagnostic VPN — Site de Planoise

  • Déplacement sur site pour reproduire et analyser le dysfonctionnement VPN
  • Symptôme : accès aux lecteurs réseau OK mais perte totale de la connexion Internet en VPN
  • Diagnostic : le trafic Internet transitait par la passerelle entreprise (tunnel intégral) au lieu de sortir localement
  • Identification de l'anomalie : métriques d'interfaces Windows incorrectes → mauvais routage
  • Développement d'un script PowerShell pour automatiser la collecte des tables de routage et métriques à distance

Analyse STP/BPDU — Cœurs de réseau

  • Investigation sur des erreurs de paquets entre les deux cœurs de réseau répartis sur deux sites
  • Révision des mécanismes Spanning Tree Protocol (BPDU, états de ports, auto-edge)
  • Diagnostic orienté vers une erreur de configuration logicielle malgré une bande passante de 6 Gb/s
  • Détection de phénomènes de flapping (instabilité de ports) sur des équipements en aval

Audit réseau terrain — Hauts de Chazal & Planoise

  • Inventaire complet des prises réseau murales (RJ45) sur le site Hauts de Chazal (dépôt tram)
  • Relevé de localisation, étiquetage et état de chaque prise dans toutes les zones
  • Identification des prises actives légitimes vs prises inutilisées à désactiver
  • Application de la même méthodologie sur le site de Planoise (dépôt bus)
  • Désactivation des ports inutilisés côté switch → neutralisation du risque d'intrusion physique
  • Documentation consolidée multi-sites conforme aux référentiels ISO 27001 / ANSSI

Enjeux identifiés

  • Un port réseau actif non surveillé = vecteur d'intrusion réel (branchement d'un équipement non autorisé)
  • Complexité du diagnostic STP : phénomènes de flapping masquant la cause racine
  • VPN : mauvais routage causé par les métriques Windows, difficile à diagnostiquer sans script dédié

Résultats obtenus

  • Cartographie complète des prises réseau sur 2 sites
  • Ports inutilisés désactivés → surface d'attaque réduite
  • Script PowerShell de collecte réseau livré pour faciliter les diagnostics futurs
  • Cause du dysfonctionnement VPN identifiée et documentée

04

Infrastructure et haute disponibilité

Renforcement de la résilience de l'infrastructure réseau et lancement du projet de sécurité physique. Cette mission regroupe les travaux visant à garantir la continuité de service et à préparer l'autonomie administrative du pare-feu local.

Projet Sécurité Intrusion — Lancement

  • Démarrage d'un projet stratégique de refonte du contrôle d'accès physique de l'entreprise
  • Inventaire exhaustif des équipements existants sur tous les sites :
  • Contrôleurs d'accès (lecteurs de badges, claviers à code)
  • Détecteurs d'intrusion (capteurs de mouvement, contacts portes/fenêtres)
  • Centrale d'alarme et équipements de supervision
  • Objectif : cartographie complète pour préparer la migration (nommage, remplacement, reconfiguration)

Fibre optique redondante — Failover FAI

  • Suivi de l'installation d'une nouvelle ligne fibre optique SFR en lien de secours
  • Objectif : haute disponibilité réseau en cas de coupure du lien principal
  • Mise en place d'un mécanisme de Failover automatique entre les deux FAI
  • Double fournisseur d'accès Internet pour garantir la continuité de service

Architecture FortiGate — VDOM

  • Problématique : gestion des règles pare-feu centralisée par la DSI du Groupe → manque de réactivité locale
  • Réunion stratégique avec le Groupe pour récupérer l'autonomie d'administration locale
  • Deux solutions étudiées : pare-feu d'interconnexion dédié vs VDOM (Virtual Domain)
  • Solution privilégiée : création d'un VDOM sur le FortiGate existant
  • Avantage : partitionner virtuellement le pare-feu sans impacter le reste du groupe

Défis rencontrés

  • Dépendance vis-à-vis de la DSI groupe pour toute modification des règles pare-feu
  • Nécessité de négocier avec le groupe pour obtenir une délégation d'administration
  • Inventaire physique complexe sur plusieurs sites distants

Résultats obtenus

  • Lien de secours fibre opératif → haute disponibilité réseau assurée
  • Architecture VDOM validée techniquement → autonomie locale en cours d'obtention
  • Cartographie complète des équipements de contrôle d'accès physique sur tous les sites

Administration, sécurité et support

Missions transversales d'administration des outils de sécurité périmétrique et d'affichage dynamique, ainsi que du support Helpdesk quotidien auprès des utilisateurs et des prestataires.

Administration FortiGate — Interception & RBAC

  • Personnalisation de la page d'interception FortiGate affichée lors de l'accès à un site bloqué
  • Audit complet des comptes utilisateurs ayant accès à la base de données de billettique (Kuba)
  • Création d'un tableau de suivi avec les dates de dernière connexion
  • Nettoyage des comptes inactifs pour renforcer la sécurité du système billettique
  • Création d'un compte de service dédié sur les switchs avec le rôle Manager (RBAC)

Samsung MagicINFO — Résolution d'incidents

  • Retour support Samsung suite au ticket ouvert lors de la migration Windows Server 2025
  • Diagnostic : conflit de versions de licences sur le serveur MagicINFO (une licence par écran)
  • Résolution : fusion des licences par l'ingénieur support → licence unique pour 6 écrans
  • Résolution d'un incident bloquant lors d'une montée de version :
  • Analyse des logs → utilisateur d'exécution sans droits d'écriture sur une table BDD
  • Élévation des privilèges sur la base de données → mise à jour validée avec succès

Support Helpdesk & prestataires

  • Journée dédiée au support de proximité et à distance en binôme avec le maître de stage
  • Résolution d'incident sur un vidéoprojecteur (paramètres d'affichage par défaut modifiés)
  • Diagnostic interphone site Planoise : problème organisationnel (visiteurs appelaient le mauvais poste)
  • Accompagnement d'un prestataire pour le déploiement d'une mallette de diagnostic bus

Difficultés rencontrées

  • Conflit de licences MagicINFO non documenté lors de la migration → nécessite l'intervention du support Samsung
  • Droits BDD insuffisants bloquant la mise à jour MagicINFO → diagnostic uniquement via les logs
  • Principe de moindre privilège : le rôle "Operator" insuffisant pour extraire les running-config des switchs

Résultats obtenus

  • Service MagicINFO stabilisé avec licence unifiée sur 6 écrans
  • Montée de version MagicINFO réussie après élévation des droits BDD
  • Base de données Kuba sécurisée (comptes inactifs supprimés)
  • Incidents utilisateurs résolus dans la journée
